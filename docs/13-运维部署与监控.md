# 13 运维部署与监控

## 13.1 部署拓扑建议

```
                    ┌─────────────┐
  客户端 ──►  CDN/WAF ──► 反向代理(Nginx) ──► Gunicorn/uWSGI ──► Django
                              │                        │
                              │                        └──► Redis
                              │
                              └──► 静态: Vue dist / 管理静态资源
```

- **Django 应用**：多 Worker 水平扩展时，共享 **同一 Redis** 实例或集群，确保情报、封禁、限流、地理池、熔断状态一致。  
- **地理池同步**：建议 **单定时任务**（或带分布式锁的任务）执行 `sync_geo_ip_pools`，避免多机同时写造成抖动（虽最终一致，但会产生重复下载）。

---

## 13.2 反向代理（Nginx）要点

- 将 **`/ip-guard/`** 转发至应用 upstream。  
- 若需正确解析客户端 IP：配置 `X-Forwarded-For` / `X-Real-IP`，并在 Django 侧配置 **`IP_GUARD_TRUSTED_PROXY_CIDRS`** 仅信任内网反代网段。  
- 超时与 body 大小：导出 CSV 或大数据接口时适当放宽 **`proxy_read_timeout`**。

---

## 13.3 Redis 高可用

- 生产建议使用 **主从 + 哨兵** 或 **Redis Cluster**。  
- 关注 **内存上限** 与 **键驱逐策略**（`maxmemory-policy`）：情报与地理池键建议 **volatile-ttl** 或明确不用 LRU 误删持久键——地理池键默认无 TTL，需避免 `allkeys-lru` 直接淘汰关键数据。  
- 定期 **RDB/AOF 备份** 以便灾难恢复（封禁与情报可重建，但审计库在 PostgreSQL/MySQL 侧另备）。

---

## 13.4 健康检查与监控指标

### 13.4.1 接口

`GET /ip-guard/api/health/`（需 staff + `view_ipguardpolicy`）返回：

- `redis_ok`、`redis_latency_ms`  
- `provider_circuit_failures`  
- `geo_ip_pools`：各池上次成功时间与行数等摘要  

可纳入 **Kubernetes liveness/readiness** 或监控系统 HTTP 检查（注意权限与认证，通常用内部探针账号）。

### 13.4.2 建议采集的业务指标

| 指标 | 来源 |
|------|------|
| 请求拦截率 | 日志或 `IpAccessLog` 聚合 |
| Provider 失败次数 | Redis 熔断键或应用日志 |
| 限流触发次数 | 可在后续版本增加计数器；当前可从审计 reason 子串统计 |
| 地理池上次同步时间 | `IpGeoPoolStatus` 或 health 接口 |

---

## 13.5 日志与告警

- 应用日志：关注 `Provider 熔断`、`地理 IP 池同步失败`、`Redis 连接异常`。  
- 告警阈值示例：  
  - `redis_ok == false` 连续 3 次  
  - `provider_circuit_failures` ≥ 配置阈值  
  - 地理池 `last_ok_at` 超过 48 小时未更新（若业务要求日更）

---

## 13.6 安全加固清单（运维向）

- 管理 API 仅内网或 VPN 可达，或前置 **mTLS / SSO**。  
- **JWT 密钥**独立于 `SECRET_KEY` 轮换。  
- 关闭不必要的 Django Admin 对外暴露或改强认证。  
- 定期轮换 **Provider API Key** 与 **Redis 密码**。
