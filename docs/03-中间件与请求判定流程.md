# 03 中间件与请求判定流程

## 3.1 入口类

- 文件：`django_ip_safeguard/middleware.py`  
- 类：`IpGuardMiddleware`  
- 配置入口：`get_settings()` → `load_effective_policy(...)` 得到**运行时策略** `runtime_config`。

若 `runtime_config.enabled` 为 `False`，中间件直接 `return self.get_response(request)`，**不参与**任何判定。

---

## 3.2 判定顺序（与代码一致）

对单次请求，在 `enabled=True` 且能解析出 `client_ip` 时，顺序如下：

1. **解析客户端 IP**  
   - `resolve_client_ip(request, trusted_proxy_cidrs)`  
   - 无 IP 则放行（避免误伤异常请求）。

2. **IP 白名单**  
   - `first_matching_rule(client_ip, ip_whitelist)`  
   - 命中：**直接放行**，不再执行黑名单、地理池、限流、封禁、情报、风险引擎。

3. **IP 黑名单**  
   - 命中：返回 `JsonResponse`，HTTP 状态码为 `block_status_code`（默认 403），体中带 `reason`。  
   - 可选写审计（`use_db_log`）。

4. **中国内 / 国际 CIDR 池规则**  
   - `evaluate_geo_ip_pool_rules(client_ip, runtime_config, cache_service)`  
   - 若 Redis 中对应池**无有效索引**（未同步或为空），该步**跳过**（不拦截），避免全站不可用。  
   - 规则含义见 [06-地理IP池与定时同步](./06-地理IP池与定时同步.md)。

5. **单 IP 每分钟请求上限**  
   - `cache_service.is_rate_limited(client_ip, rate_limit_per_minute)`  
   - `rate_limit_per_minute <= 0` 表示关闭。  
   - 实现为 Redis 计数 + 60 秒窗口，详见 [08-Redis缓存封禁限流与熔断](./08-Redis缓存封禁限流与熔断.md)。

6. **封禁缓存**  
   - 命中：直接 JSON 拦截（不写情报）。

7. **情报缓存 / Provider**  
   - 先读 Redis 情报缓存；未命中则判断是否熔断、是否获取去重锁，再调用 `provider.fetch_ip_intel`。  
   - 失败时走 `_handle_provider_failed`（fail-open / fail-close 与路径前缀）。

8. **风险引擎**  
   - `evaluate_ip_risk(ip_intel, runtime_config)`  
   - 不通过时可按决策触发 `ban_ip`（写 Redis 封禁）。

9. **放行**  
   - `get_response(request)`；可选写「放行」审计。

---

## 3.3 流程示意（文本）

```
请求进入
   │
   ▼
enabled? ──否──► 放行（后续中间件/视图）
   │
  是
   │
解析 IP ──失败──► 放行
   │
白名单命中? ──是──► 放行
   │
黑名单命中? ──是──► 拦截
   │
地理池规则拦截? ──是──► 拦截
   │
限流触发? ──是──► 拦截
   │
封禁命中? ──是──► 拦截
   │
情报（缓存/Provider）
   │
风险引擎 ──拦截──► 可选封禁 + 拦截
   │
放行
```

---

## 3.4 设计取舍说明

| 取舍 | 说明 |
|------|------|
| 白名单最先 | 办公网、健康检查、监控探针等应无条件绕过昂贵逻辑。 |
| 黑名单在情报前 | 已知恶意网段无需再调外部情报，降本与时延。 |
| 地理池在限流前 | 先按「地域/列表」挡掉，再对剩余流量做频控。 |
| 池空不生效 | 防止未跑同步任务导致「仅允许池内」误拦全球。 |
| JSON 拦截体 | 便于 API 与前后端分离；若需 HTML 友好页需自行在中间件外包一层或改扩展点（当前未提供模板钩子）。 |

---

## 3.5 扩展与定制建议

- **新增判定阶段**：建议 fork 中间件或在同一中间件内按顺序插入，注意保持**幂等**与**可观测**（审计 reason）。  
- **与业务用户绑定**：当前中间件不读取 `request.user`；若需「已登录用户跳过」，应在更后中间件或视图中处理，或扩展本中间件（注意性能）。
