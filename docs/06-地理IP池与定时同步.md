# 06 地理 IP 池与定时同步

## 6.1 功能目标

在**不依赖商业 GeoIP 库**的前提下，通过维护一份（或多份）**公开或自建的 CIDR 文本列表**，在 Django 请求前置阶段做：

- **仅允许列表内访问**（典型：仅允许「中国内」地址段）；或  
- **列表内一律拦截**（典型：封禁已知的「国际」或恶意托管网段集合）。

系统维护两套逻辑池键名：

| 池键 | Redis 数据键 | 说明 |
|------|----------------|------|
| `china` | `ip_guard:geo_pool:data:china` | 默认从可配置的 GitHub `china_ip_list` 拉取 |
| `international` | `ip_guard:geo_pool:data:international` | 需配置 `IP_GUARD_GEO_INTERNATIONAL_POOL_URL` 才会同步 |

---

## 6.2 数据源格式（重要）

远程 URL 返回 **纯文本**，**每行一条**：

- IPv4 / IPv6 的 **CIDR**（如 `1.2.3.0/24`）；或  
- 单个 IP（如 `203.0.113.1`）。

支持以 `#` 或 `;` 开头的注释行；空行忽略。

**国际池**没有内置默认 URL：需你自行托管或选用合规的第三方列表（注意版权与更新频率），填入 `IP_GUARD_GEO_INTERNATIONAL_POOL_URL`。

---

## 6.3 规则语义

策略字段（数据库 `IpGuardPolicy` 或 settings 默认项）：

| 取值 | 含义 |
|------|------|
| `off` | 不启用该池的匹配逻辑。 |
| `allow_only_in_pool` | 客户端 IP **必须落在**该池 CIDR 集合内，否则拦截。 |
| `block_in_pool` | 客户端 IP **若落在**该池 CIDR 集合内，则拦截。 |

两套池相互独立：可同时配置「中国池 allow_only」与「国际池 block」等组合（需理解业务含义，避免逻辑矛盾）。

---

## 6.4 「池未就绪」时的行为

若 Redis 中对应池 JSON **不存在**或 **无任何有效 v4/v6 数据**，中间件中该池相关规则 **不生效**（跳过，不拦截）。  

目的：避免未执行同步任务时出现「allow_only + 空池 = 全员不可访问」的事故。

**上线建议顺序**：

1. 配置 URL（至少中国池默认已有）。  
2. 执行一次 `python manage.py sync_geo_ip_pools` 或调用 API `POST /ip-guard/api/geo-pools/sync/`。  
3. 确认 `GET /ip-guard/api/geo-pools/status/` 或健康检查中 `geo_ip_pools` 有成功时间与行数。  
4. 再在策略中心打开 `allow_only_in_pool` 等规则。

---

## 6.5 索引结构与性能

- IPv4：所有网段合并为**有序不相交区间**，查询使用二分（`bisect`）。  
- IPv6：保留网段字符串列表，查询时对列表逐项 `ip in network`（一般中国列表以 v4 为主）。  
- Worker 侧对 Redis 原始 JSON 有 **进程内短缓存**（`IP_GUARD_GEO_POOL_INDEX_CACHE_SECONDS`，默认 60 秒），减轻 Redis QPS。

实现文件：

- `services/geo_ip_pool_index.py`：解析与合并  
- `services/geo_ip_pool_runtime.py`：缓存与规则判定  
- `services/geo_ip_pool_sync.py`：HTTP 拉取与写 Redis、更新 `IpGeoPoolStatus`

---

## 6.6 同步方式

### 6.6.1 管理命令（推荐用于 crontab）

```bash
python manage.py sync_geo_ip_pools
```

行为：

- 始终尝试同步 **中国池**（URL 来自 `IP_GUARD_GEO_CHINA_POOL_URL`）。  
- 若 **`IP_GUARD_GEO_INTERNATIONAL_POOL_URL` 非空**，同步国际池；否则跳过（**不删除** Redis 里已有的国际池数据）。

### 6.6.2 管理 API（控制台「立即同步池」）

- `POST /ip-guard/api/geo-pools/sync/`  
- 权限：`django_ip_safeguard.change_ipguardpolicy`  
- 需携带 CSRF（与 Session 或 Cookie 场景一致）；JWT 用户同样需要有效 CSRF（见认证文档）。

### 6.6.3 定时任务示例

**crontab**（每日 3 点）：

```cron
0 3 * * * cd /srv/app && /srv/venv/bin/python manage.py sync_geo_ip_pools >> /var/log/ip_guard_geo_sync.log 2>&1
```

**systemd timer**：将 `ExecStart` 指向同上命令即可。

## 6.7 数据库表：`IpGeoPoolStatus`

| 字段 | 说明 |
|------|------|
| `pool_key` | `china` / `international` |
| `source_url` | 最近一次同步使用的 URL（截断存储） |
| `line_count` | 文本非空行计数（近似） |
| `v4_interval_count` | 合并后 IPv4 区间数 |
| `v6_net_count` | IPv6 网段条数 |
| `last_ok_at` | 上次成功时间 |
| `last_error` | 上次失败信息 |

在 Django Admin 中为**只读展示**用途。

---

## 6.8 故障排查

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 规则开了但不生效 | Redis 无数据或池为空 | 执行同步，查 status |
| 同步失败 | 网络出网、TLS、URL 404 | 看 `last_error`，换镜像或自建 |
| 误判多 | 列表过旧或含异常段 | 更换数据源或自建裁剪列表 |
| 多 Worker 规则不一致 | 进程内缓存 TTL 内旧数据 | 等待 TTL 或重启进程；同步后会清本地缓存 |
